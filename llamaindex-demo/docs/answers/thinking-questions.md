## 第 1 题：如果不用 LlamaIndex，怎么实现 RAG？

**问题**：假设你只能用 Python 标准库和 OpenAI API，
从头实现一个最简单的 RAG 系统。你需要哪些组件？各做什么？

<details>
<summary>点击查看参考答案</summary>

你需要 4 个组件：

1. **文档加载器**：读取 PDF/TXT/MD 等文件 → `open().read()`
2. **文本切分器**：把长文档切成小块 → 按句子/段落切，控制每块 500 字左右
3. **向量化**：调用嵌入 API 把文本块转成向量 → `openai.Embedding.create()`
4. **检索 + 生成**：用户问题 → 向量化 → 计算余弦相似度 → 取 Top-K 拼接成 prompt → 调用 LLM 生成

这就是 LlamaIndex 帮我们做的——它把这些步骤封装成优雅的抽象。

</details>

## 第 2 题：为什么 chunk_overlap 很重要？给一个具体场景。

**问题**：假设有一段话：「Transformer 架构由 Vaswani 等人于 2017 年
提出。它基于自注意力机制，摒弃了循环结构。」如果 chunk_size=30 且 overlap=0，
会发生什么？

<details>
<summary>点击查看参考答案</summary>

切分结果可能是：
- Chunk 1: 「Transformer 架构由 Vaswani 等人于 2017 年提出。」
- Chunk 2: 「它基于自注意力机制，摒弃了循环结构。」

问题：「它」指代什么？Chunk 2 脱离了上下文，「它」失去了指代对象。
LLM 检索到 Chunk 2 时，无法确定「它」是 Transformer。

有 overlap：Chunk 1 的末尾会出现在 Chunk 2 的开头，保留了上下文连贯性。

</details>

## 第 3 题：什么时候用向量检索，什么时候用关键词检索？

**问题**：用户查询「RAG 论文中第 3.2 节的实现细节」。
纯向量检索和纯关键词检索各有什么问题？怎么解决？

<details>
<summary>点击查看参考答案</summary>

- **纯向量检索**：「3.2」「第」这些词在语义空间中意义不大，向量可能找不到精确匹配。
- **纯关键词检索**：「实现细节」这个词可能不直接出现在文档中（文档可能写的是「具体做法」「技术方案」），关键词匹配会遗漏。

**解决方案：混合检索（Hybrid Search）**
向量负责理解语义（「实现细节」≈「具体做法」），关键词负责精确匹配（「3.2 节」）。
两者分数加权合并，结果最准确。

</details>

## 第 4 题：总结 compact vs refine vs tree_summarize 的适用场景。

**问题**：各给出一个最适合的场景。

<details>
<summary>点击查看参考答案</summary>

| 模式 | 最适合的场景 | 原因 |
|------|-------------|------|
| compact | 检索到 3-5 个片段，答案需要综合这些片段 | 一次打包，速度快，token 省 |
| refine | 检索到多个片段各自包含不同信息，需要逐步融合 | 逐一消化，避免信息遗漏 |
| tree_summarize | 检索到 10+ 个片段，内容多到一条消息装不下 | 分层总结，控制单次 token 量 |

</details>

## 第 5 题：ChatEngine 怎么处理代词指代？

**问题**：用户第一轮问「什么是 RAG？」第二轮问「它有什么缺点？」
ChatEngine 怎么知道「它」指的是 RAG？

<details>
<summary>点击查看参考答案</summary>

这是 `CondenseQuestionChatEngine` 的核心机制：

1. LLM 看到对话历史：「什么是 RAG？」→「RAG 是...」
2. 用户新问题：「它有什么缺点？」
3. LLM 被要求「把对话历史 + 新问题压缩成独立问题」
4. LLM 输出：「RAG（检索增强生成）有什么缺点？」
5. 用这个独立问题去检索 → 精准！

这就叫「问题压缩」（Question Condensation）——解决了代词消解问题。

</details>

## 第 6 题：FunctionTool vs QueryEngineTool 怎么选？

**问题**：你需要一个 Agent，它既能查天气（调用 API），又能查技术文档（RAG）。
应该各用什么 Tool？

<details>
<summary>点击查看参考答案</summary>

- **天气查询** → `FunctionTool`：封装一个调用天气 API 的 Python 函数。这是「确定性的功能调用」。
- **技术文档查询** → `QueryEngineTool`：封装一个 VectorStoreIndex 的查询引擎。这是「语义检索」。

选择原则：
- 函数调用（API/计算/数据库）→ FunctionTool
- 文档检索（RAG/知识库）→ QueryEngineTool

Agent 会根据问题自动选正确的工具——这就是 Agent 的核心价值。

</details>

## 第 7 题：什么时候该持久化索引？

**问题**：你有一个包含 5000 篇技术文档的知识库。每次启动都重索引，
和只索引一次然后持久化，差别有多大？

<details>
<summary>点击查看参考答案</summary>

**重索引（不持久化）**：
- 每次启动：5000 篇 × 嵌入调用 × $0.00002/1K tokens ≈ 每次成本可观
- 每次启动：等待时间 5-30 分钟（取决于文档量和 API 速率）
- 完全浪费

**持久化后**：
- 首次：同上（一次性成本）
- 后续启动：直接加载文件 → 秒级完成 → 零额外费用
- 只在文档更新时重新索引

结论：只要不是一次性实验，都应该持久化。

</details>

## 第 8 题：RAG 的「幻觉」从哪来？怎么减少？

**问题**：RAG 号称能减少幻觉，但有时 RAG 系统仍然会编造信息。
列举 3 个原因和对应解决方案。

<details>
<summary>点击查看参考答案</summary>

1. **检索不相关**：
   - 原因：检索到的内容跟问题无关，LLM 只能「硬编」
   - 解决：改进检索（混合检索、reranker、查询重写）

2. **LLM 忽视检索结果**：
   - 原因：LLM 「自信」地使用自己的训练知识，不理会检索结果
   - 解决：在 prompt 中强调「只基于参考资料回答，不知道就说不知道」

3. **检索到的内容本身有误**：
   - 原因：文档库中有错误信息
   - 解决：文档质量管控，定期审核知识库

</details>

## 第 9 题：评估 RAG 系统该看哪些指标？

**问题**：你上线了一个 RAG 系统，老板问「效果怎么样」。
你该报告哪些指标？

<details>
<summary>点击查看参考答案</summary>

**检索侧指标**（找得准不准）：
- **Hit Rate**：正确答案是否在 Top-K 结果中
- **MRR (Mean Reciprocal Rank)**：正确答案排第几
- **NDCG**：考虑排名位置的相关性打分

**生成侧指标**（答得好不好）：
- **Faithfulness（忠实度）**：答案是否基于检索内容（有没有编造）
- **Answer Relevancy（答案相关性）**：答案是否切题
- **Context Relevancy（上下文相关性）**：检索到的内容是否必要

**业务指标**（用户满不满意）：
- 用户反馈（👍/👎）
- 任务完成率
- 回复率（用户追问比例）

</details>

## 第 10 题：递归检索（Recursive Retrieval）适合什么场景？

**问题**：你有一个 500 页的技术手册（章→节→段→句 的层次结构）。
用户可能问「第三章讲了什么」也可能问「XX 函数的返回值类型是什么」。
单一粒度的检索有什么问题？

<details>
<summary>点击查看参考答案</summary>

单一粒度的问题：
- 如只用「节」级别检索：问「第三章」能回答，但问「XX 函数返回值」粒度太粗，不精准。
- 如只用「句」级别检索：问「XX 函数」很精准，但问「第三章概述」需要聚合大量句子。

**递归检索的解决方案**：
1. 建立层次索引（父节点 = 概括，子节点 = 细节）
2. 查询时先检索父节点（粗筛）
3. 在匹配的父节点下再检索子节点（精筛）
4. 用子节点的细节内容生成答案

这样无论查询粒度粗细，都能找到合适的内容。

</details>

## 第 11 题：NodePostprocessor 和 Reranker 有什么区别？

**问题**：两者都在检索后运行，都用于改进结果质量。它们是一回事吗？

<details>
<summary>点击查看参考答案</summary>

不是一回事，但有重叠。

**NodePostprocessor** 是一个更广泛的概念：
- 可以做的事：过滤低分节点、去重、替换文本、重排序...
- SimilarityPostprocessor、KeywordNodePostprocessor 都是 Postprocessor
- Reranker 可以被实现为一个 Postprocessor

**Reranker** 是专门做「重新打分」的：
- 输入：检索到的 Top-K 节点
- 输出：重新排序后的节点（分数更新）
- 使用专门的 Cross-Encoder 模型（比 Bi-Encoder 更准但更慢）

流程通常是：检索（快 + 多）→ Postprocessor 过滤（去噪）→ Reranker 重排（精准）。

</details>

## 第 12 题：如果不做 RAG，还有哪些让 LLM 获取知识的方法？

**问题**：对比 RAG、Fine-tuning、Prompt Engineering 三种方法，
各适用于什么场景？

<details>
<summary>点击查看参考答案</summary>

| 方法 | 原理 | 适用场景 | 优缺点 |
|------|------|---------|--------|
| Prompt Engineering | 把知识直接写在 prompt 里 | 少量知识、一次性任务 | 简单但不适合大量知识 |
| RAG | 检索外部知识库 | 海量文档、知识频繁更新 | 灵活但增加系统复杂度 |
| Fine-tuning | 用知识数据训练模型 | 风格/格式学习、领域适应 | 高质量但昂贵、不灵活 |

最佳实践：
- 知识量大 + 更新频繁 → RAG
- 需要改 LLM 的行为/格式 → Fine-tuning
- 两者结合（RAG + Fine-tuned 模型）→ 最优
- 少量一次性知识 → Prompt 直接写进去

</details>
