# 多智能体协作系统 —— 实现原理

## 1. 核心架构

本系统采用 **编排式多智能体架构（Orchestrated Multi-Agent Architecture）**，由一个中央工作流引擎负责任务分发和流程控制，各 Agent 在指定的阶段执行特定角色任务。

```
┌─────────────────────────────────────────────────────┐
│                  用户接口层                           │
│            main.py (CLI / Rich UI)                   │
└────────────────────┬────────────────────────────────┘
                     │
┌────────────────────▼────────────────────────────────┐
│                 工作流编排层                          │
│   LangGraph StateGraph (节点、边、状态管理、条件路由)  │
└────────────────────┬────────────────────────────────┘
                     │
┌────────────────────▼────────────────────────────────┐
│                   Agent 层                           │
│  ┌──────┐  ┌──────┐  ┌──────┐  ┌──────┐  ┌──────┐ │
│  │研究   │→│分析   │→│写作   │→│审核   │→│汇总   │ │
│  └──────┘  └──────┘  └──────┘  └──────┘  └──────┘ │
└────────────────────┬────────────────────────────────┘
                     │
┌────────────────────▼────────────────────────────────┐
│                  工具与基础设施层                      │
│  共享上下文(黑板) │ 知识库 │ 计算引擎 │ LLM 配置      │
└─────────────────────────────────────────────────────┘
```

## 2. 关键设计模式

### 2.1 黑板模式（Blackboard Pattern）

多个 Agent 不直接通信，而是通过一个共享的"黑板"（Shared Context）读写数据。这种模式的优点是：

- **解耦**：Agent 之间没有直接依赖，每个 Agent 只关心自己的输入和输出
- **可追溯**：所有中间结果都保存在黑板上，便于调试和审计
- **灵活性**：可以方便地插入、移除或替换 Agent，不影响整体架构

```python
# 共享上下文 —— 所有 Agent 通过字典进行数据交换
shared_context = {
    "topic": "",
    "research_notes": "",      # 研究 Agent 产出
    "analysis_result": "",     # 分析 Agent 产出
    "draft_content": "",       # 写作 Agent 产出
    "review_feedback": "",     # 审核 Agent 产出
    "final_output": "",        # 汇总 Agent 产出
    "status": "initialized",   # 流程状态
}
```

### 2.2 提示词工程（Prompt Engineering）

每个 Agent 通过精心设计的 **系统提示词（System Prompt）** 获得角色定义和行为指南。提示词包含：

- **角色定义**：你是谁，你的专业领域
- **职责说明**：你需要完成的具体任务
- **行为规范**：你应该如何思考和工作
- **输出格式**：你应该以什么格式输出

### 2.3 工具绑定（Tool Binding）

Agent 可以通过 LLM 的 function calling 能力调用外部工具。本系统实现了：

- **知识库检索**：模拟从知识库获取信息
- **数学计算**：执行数值运算
- **共享上下文读写**：Agent 间的数据交换通道

## 3. 技术实现细节

### 3.1 LangGraph 状态图

使用 LangGraph 的 `StateGraph` 构建工作流：

```python
workflow = StateGraph(AgentState)

# 添加节点（每个节点对应一个 Agent 的执行）
workflow.add_node("research", research_node)
workflow.add_node("analysis", analysis_node)
workflow.add_node("writing", writing_node)
workflow.add_node("review", review_node)
workflow.add_node("summarize", summarize_node)

# 添加边（定义执行顺序）
workflow.add_edge("research", "analysis")
workflow.add_edge("analysis", "writing")
workflow.add_edge("writing", "review")
workflow.add_edge("review", "summarize")
workflow.add_edge("summarize", END)

workflow.set_entry_point("research")
```

### 3.2 Agent 的执行模式

每个 Agent 的执行遵循统一的模式：

1. **读取输入**：从共享上下文读取所需数据
2. **更新状态**：将状态标记为"进行中"
3. **调用 LLM**：使用 System Prompt + User Message 调用大语言模型
4. **处理输出**：可能涉及工具调用（Tool Calls）的循环
5. **写入结果**：将输出写入共享上下文
6. **更新状态**：将状态标记为"已完成"

### 3.3 错误处理机制

工作流内置了错误传播机制：

- 每个节点在 `try/except` 块中执行
- 发生错误时，将错误信息写入状态并标记为 `error`
- 条件路由 `should_continue()` 检测错误状态并终止流程
- 错误不会导致整个程序崩溃，而是优雅地报告失败原因

## 4. 与传统架构对比

| 特性 | 单体 Agent | 多 Agent 编排（本方案） |
|------|-----------|----------------------|
| 任务粒度 | 单 Agent 全栈处理 | 多 Agent 分工协作 |
| 专注度 | 提示词过长时注意力分散 | 每个 Agent 专注单一角色 |
| 可维护性 | 提示词耦合度高 | 职责分离，易于维护 |
| 可扩展性 | 新增能力需重构提示词 | 新增 Agent 即可扩展 |
| 可观测性 | 黑盒输出 | 每个阶段产出可独立审查 |
| 质量保证 | 依赖单次输出 | 有专门审核环节把关 |
