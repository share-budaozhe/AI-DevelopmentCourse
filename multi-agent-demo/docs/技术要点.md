# 多智能体协作系统 —— 技术要点

## 1. LangGraph 状态图详解

### 1.1 StateGraph 核心概念

| 概念 | 说明 | 本 Demo 中的应用 |
|------|------|-----------------|
| **State** | 工作流的全局状态 | `AgentState` TypedDict，包含课题、各阶段产出、状态等 |
| **Node** | 工作流中的处理节点 | 每个 Agent 对应一个节点（research/analysis/writing/review/summarize）|
| **Edge** | 节点间的连接 | 顺序边（research→analysis→...）、条件边（错误检测）|
| **Entry Point** | 工作流入口节点 | research 节点 |

### 1.2 状态管理的设计考量

```python
class AgentState(TypedDict):
    """每个字段代表协作流程中的一个里程碑"""
    topic: str           # 输入：用户课题
    writing_style: str   # 输入：文章风格
    research_output: str  # 里程碑1：研究产出
    analysis_output: str  # 里程碑2：分析产出
    draft_output: str     # 里程碑3：初稿
    review_output: str    # 里程碑4：审核意见
    final_output: str     # 里程碑5：最终交付物
    status: str           # 流程状态
    error: str            # 错误信息
```

**设计原则**：
- 状态字段应当反映业务里程碑而非技术实现细节
- 使用 TypedDict 获得类型提示支持
- 为错误处理预留专用字段

## 2. Agent 设计要点

### 2.1 提示词设计原则

```
高质量 System Prompt 的结构：
┌─────────────────────────────────────┐
│ 1. 角色定义  "你是一个专业的XX"       │
│ 2. 职责说明  "你的职责是：1... 2..." │
│ 3. 行为规范  "你注重XX，擅长XX"       │
│ 4. 输出格式  "请使用Markdown格式"     │
│ 5. 风格约束  "请始终用中文输出"       │
└─────────────────────────────────────┘
```

### 2.2 Agent 粒度控制

Agent 的角色定义应当遵循 **单一职责原则**：

| 粒度 | 说明 | 示例 |
|------|------|------|
| 过粗 | 提示词过长，LLM 注意力分散 | "请完成研究、分析、写作、审核和汇总" |
| 适中 ✅ | 职责明确，提示词聚焦 | "你是研究员，负责信息搜集与整理" |
| 过细 | 节点过多，流程管理复杂 | "分为检索Agent、摘要Agent、分类Agent..." |

### 2.3 Temperature 参数调优

不同 Agent 角色对创造性要求不同，应设置不同的 temperature：

| Agent | Temperature | 原因 |
|-------|-------------|------|
| 研究 Agent | 0.3 | 需要准确性和事实性，创造性要求低 |
| 分析 Agent | 0.4 | 需要适度创造性进行洞察 |
| 写作 Agent | 0.7 | 需要较高的创造性进行内容创作 |
| 审核 Agent | 0.3 | 需要严谨性和一致性 |
| 汇总 Agent | 0.5 | 在准确性和创造性之间平衡 |

## 3. 工具绑定机制

### 3.1 Function Calling 流程

```
LLM 收到用户消息
    │
    ├─ LLM 判断是否需要调用工具
    │     │
    │     ├─ 需要 → 返回 tool_calls 结构
    │     │         │
    │     │         ▼
    │     │      执行工具函数
    │     │         │
    │     │         ▼
    │     │      将工具结果返回给 LLM
    │     │         │
    │     │         ▼
    │     │      LLM 生成最终回复
    │     │
    │     └─ 不需要 → 直接生成回复
```

### 3.2 工具定义方式

使用 LangChain 的 `@tool` 装饰器定义工具：

```python
from langchain_core.tools import tool

@tool
def search_knowledge_base(query: str) -> str:
    """根据关键词检索知识库"""
    # 函数文档字符串会作为工具描述传给 LLM
    # 参数类型注解用于生成工具参数 Schema
    ...
```

**关键要点**：
- 函数名和参数名要语义化，LLM 通过名称理解工具用途
- 文档字符串（docstring）是工具描述，直接影响 LLM 是否选择调用
- 类型注解用于生成 JSON Schema，确保参数格式正确

## 4. 共享上下文设计

### 4.1 为什么用共享内存而非数据库

| 考量 | 共享内存 | 数据库 |
|------|---------|--------|
| 速度 | 纳秒级 | 毫秒级 |
| 复杂度 | 零配置 | 需要连接池、迁移等 |
| 粒度 | 适用 Demo/原型 | 适用生产系统 |
| 调试 | 直接查看内存 | 需要查询 |

### 4.2 扩展为持久化方案

本 Demo 的共享上下文可轻松扩展为：

- **文件持久化**：将 shared_context 序列化为 JSON 文件
- **Redis 缓存**：使用 Redis Hash 存储，支持分布式 Agent
- **数据库**：使用关系型数据库记录每次协作的完整轨迹

## 5. 配置管理

### 5.1 多 Provider 支持

系统通过环境变量支持灵活的 LLM 提供商切换：

```python
def get_llm():
    provider = os.getenv("LLM_PROVIDER", "openai")
    if provider == "deepseek":
        # 使用 DeepSeek
    else:
        # 使用 OpenAI
```

### 5.2 配置优先级

```
环境变量 (.env) > 代码默认值 > 函数参数
```

## 6. 可观测性设计

### 6.1 日志与输出

使用 Rich 库实现结构化的终端输出：

- **Panel**：突出显示重要信息块
- **Table**：结构化展示 Agent 角色和统计信息
- **Markdown**：渲染 Markdown 格式的输出
- **颜色编码**：不同 Agent 使用不同颜色标识

### 6.2 状态追踪

共享上下文中的 `status` 字段记录了完整的状态流转历史：
```
initialized → researching → research_completed
→ analyzing → analysis_completed
→ writing → writing_completed
→ reviewing → review_completed
→ summarizing → completed
```
