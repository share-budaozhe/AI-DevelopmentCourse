# RAG Demo 思考题答案

## 第二阶段：文档加载与分块

### Q1: `chunk_overlap=100` 有什么用？去掉会怎样？

**核心作用：防止关键信息被切在分块边界上。**

假设有一段文字："Python的常用库包括NumPy和Pandas。NumPy擅长数值计算。"

- `chunk_size=30, overlap=0`：第一块是"Python的常用库包括NumPy和Pandas。"，第二块是"NumPy擅长数值计算。"——两块各自完整，没问题。
- 但如果边界恰好落在关键概念中间："Python的常用库包括NumPy和Pandas。NumPy擅长数值计算，而Pandas擅长数据处理。"——第一块到"而Pandas"就截断了，"擅长数据处理"被分到下一块。

`overlap=100` 让相邻两块有 100 个字符的重叠区域，相当于在边界处加了"冗余"。这样即使用户搜"Pandas 数据处理"，两块都能命中。

**去掉会怎样：** 边界处的信息可能只在某一块中存在，另一块完全不知道，检索时容易漏掉。不过重叠会增加总块数、多占存储——本质是召回率和存储成本的 tradeoff。

---

### Q2: 如果把 `chunk_size` 从 500 改成 100，会多出多少块？

**粗略估算：块数约等于总字符数 / (chunk_size - overlap)。**

设 3 篇文档共约 2500 字：
- `chunk_size=500, overlap=100`：2500 / (500-100) ≈ 6.25 → 约 6-7 块（实际 demo 只生成 4 块，因为文档不等长且有自然段落边界）
- `chunk_size=100, overlap=100`：此时 overlap = chunk_size，每块实际新增为 0——TextSplitter 会把这种情况退化处理，但逻辑上块数会急剧增加。

一般情况下块数大概翻 5 倍。块太小的问题：
1. 单块信息量不足，LLM 拿到的上下文碎片化
2. 向量库膨胀，检索变慢
3. 语义不完整——"机器学习是人工智能的一个" 断在半截

**经验值：** `chunk_size=300~800` 是中文字档的常见选择，配合 `overlap=chunk_size*0.1~0.2`。

---

### Q3: 分隔符顺序 `["\n\n", "\n", "。", ".", " ", ""]` 为什么这样排？

**原则：从粗到细，优先在语义自然边界处切分。**

| 分隔符 | 语义 | 切在哪 |
|--------|------|--------|
| `\n\n` | 段落边界 | 两段之间 |
| `\n` | 行边界 | 两行之间 |
| `。` / `.` | 句子边界 | 两句之间 |
| ` ` | 词边界 | 两词之间 |
| `""` | 字符级 | 任意位置 |

递归策略：代码在 `TextSplitter.split()` 中先尝试 `\n\n` 切，如果某一段仍然超过 `chunk_size`，再用 `\n` 切，以此类推。这样确保：
1. 优先在段落边界切 → 每块是完整段落
2. 段落太长就在行边界切 → 每块是完整行
3. 行太长就在句子边界切 → 每块是完整句子
4. 最终兜底到字符级强制切断

如果顺序反过来（先切空格），一段话会被切成不完整的碎片。

## 第三阶段：文本向量化

### Q1: 为什么"机器学习"在查询向量中权重高，"天气"权重低？

**因为 TF-IDF 的两个因子都在起作用：**

查询"什么是机器学习"分词后 = `["什","么","是","机","器","学","习"]`（中文逐字分解）。

**IDF 角度（语料层面）：**

"机器学习"这组字（机、器、学、习）在语料中可能只出现在 `machine_learning.txt` 和 `rag_explained.txt` 里，不在 `python_intro.txt` 中出现——它们在 3 篇文档中出现的文档数少 → IDF 高 → 区分度高。

而"今"、"天"、"气" 这组字（来自"今天天气很好"）如果只在单独一篇无关文档中出现 → IDF 也高。但关键在于：

**TF 角度（查询层面）：**

查询"什么是机器学习"中 "天" 根本没出现 → TF=0 → 最终 TF-IDF=0。查询向量中所有 "天气" 相关维度的值都是 0。

所以本质是：**TF-IDF 只对查询中实际出现的词给非零权重**，而这些词的 IDF 高低决定了哪些是"有区分力的关键词"。

---

### Q2: TF-IDF 和 BERT/sentence-transformers 产生的向量有什么本质区别？

| 维度 | TF-IDF | BERT / Sentence-Transformers |
|------|--------|------------------------------|
| 表示方式 | 稀疏向量（大部分维度为0） | 稠密向量（如 384/768 维，全为非零） |
| 语义理解 | **字面匹配**：只看词是否出现 | **语义匹配**：理解同义词和上下文 |
| 向量含义 | 第 i 维 = 词表中第 i 个词的权重 | 每个维度是人类不可解释的抽象特征 |
| "苹果很好吃" vs "这水果不错" | 相似度 ≈ 0（没有共同词） | 相似度 ≈ 0.8（语义相近） |
| "苹果很好吃" vs "苹果股价大跌" | 相似度高（都含"苹果"） | 相似度低（一个水果一个公司） |
| 依赖 | 仅 NumPy，无需模型下载 | 需要下载几百 MB 的模型 |

**本质区别：** TF-IDF 是词袋模型（bag-of-words），认为文本就是一堆词的集合，词与词之间独立。BERT 是上下文模型，同一个"苹果"在"吃苹果"和"买苹果手机"里会产生不同的向量表示。

### Q3: L2 归一化后点积为什么就等于余弦相似度？

**数学推导：**

两个向量 A 和 B 的余弦相似度：

$$\cos(\theta) = \frac{A \cdot B}{\|A\| \times \|B\|}$$

L2 归一化后：`A_norm = A / ||A||`，`B_norm = B / ||B||`，此时 `||A_norm|| = 1`，`||B_norm|| = 1`。

归一化后的点积：

$$A_{norm} \cdot B_{norm} = \frac{A}{\|A\|} \cdot \frac{B}{\|B\|} = \frac{A \cdot B}{\|A\| \times \|B\|} = \cos(\theta)$$

**所以**：对向量做 L2 归一化（让长度变成 1），直接用 `np.dot()` 算出的结果就是余弦相似度，省掉了每次检索都除两个模长的计算。

这也是 `embedder.py` 第 65 行做 L2 归一化、`store.py` 第 71 行直接 `np.dot()` 的原因。

## 第四阶段：向量存储与检索

### Q1: `top_k=5` 和 `top_k=1` 对最终答案有什么影响？

| | top_k=1 | top_k=5 |
|------|----------|------|
| LLM 看到的上下文 | 仅最相关的 1 个片段 | 5 个不同角度的片段 |
| 答案完整性 | 可能遗漏次要信息 | 更全面 |
| 噪音风险 | 低（但如果第1条不相关就GG） | 可能有低相关度的噪音混入 |
| 典型场景 | 事实性问答（"Python作者是谁"） | 综述性问答（"介绍机器学习"） |

**比如问"机器学习有哪些类型"：**
- top_k=1：可能只命中"监督学习"那段，答案里只有监督学习
- top_k=5：会覆盖监督学习、无监督学习、强化学习、深度学习等所有相关段落

**经验值：** 一般设 3~8，事实验证类设低一些，综述类设高一些。

---

### Q2: `min_score` 过滤阈值设多少合适？

**这取决于嵌入模型和数据类型，没有绝对数值。**

- TF-IDF：相似度一般在 0~0.6 之间，0.1 以下基本是噪音。可以设 `min_score=0.05` 或 `0.1`。
- Sentence-Transformers：余弦相似度通常在 0.3~0.9 之间，`min_score=0.3` 或 `0.5` 常见。

**更好的做法：不设死阈值，而是动态取 Top-K，然后看分数落差。**

像 demo 中检索"什么是机器学习"的结果：
```
[1] 0.4750  ← 第一名
[2] 0.2341  ← 第二名，骤降一半 → 结果1明显相关
[3] 0.0107  ← 第三名，降到几乎为零 → 噪音
[4] 0.0000  ← 完全无关
```

这种情况下，可以在 retriever 里加一个**相对分数过滤**：如果某条结果的分数低于第一名分数的 20%，就丢弃。这样会自动截断到前两条。

### Q3: JSON 持久化的向量库在 10 万条数据时有什么问题？

| 问题 | 说明 |
|------|------|
| **内存爆炸** | JSON 要把全部向量 load 到内存，10 万条 × 500 维 × 4 字节(float32) = 200MB，加上文本可能上 GB |
| **检索变 O(N)** | `np.dot(embeddings, query_vec)` 每次和全部 10 万条做点积，无索引加速 |
| **写入变慢** | 每次 `add()` 都全量写 JSON，`json.dump()` 序列化 200MB 数据可能需要数秒 |
| **单文件瓶颈** | 所有数据挤在一个 JSON 文件里，无法增量更新或分片 |

**这正是为什么生产环境要用 ChromaDB / FAISS / Milvus：**

- FAISS：构建 HNSW 或 IVF 索引，检索从 O(N) 降到 O(log N)
- ChromaDB：基于 SQLite，支持增量写入、元数据过滤
- Milvus：分布式，支持十亿级向量

Demo 里的 JSON 存储是教学用的，清晰展示了向量库的核心数据结构，但不适合生产规模。

## 第五阶段：答案生成

### Q1: 如果 SYSTEM_PROMPT 去掉第2条"充分利用所有参考文档"，输出会有什么变化？

当前第 2 条：
> 充分利用所有提供的参考文档，提取关键信息组织成完整、详实的回答。

**去掉后：** LLM 可能只从最相关的 1-2 个片段中提取信息，忽略排名靠后但包含补充信息的片段。典型表现就是用户遇到的那种——回答只有一句话。

这就是提示词工程的典型场景：**LLM 默认倾向于简洁**，需要显式指令告诉它"用全所有材料、结构化输出、达到一定篇幅"。不加约束的话，模型通常会选择最省 token 的方式作答。

---

### Q2: `max_tokens=2048` 设太小会截断，设太大会怎样？

**设太小（如 256）：**
回答被生生截断，可能停在半句话，丢失关键信息。

**设太大（如 16384）：**
1. **浪费钱**——按 token 计费，但实际回答可能只有 500 tokens
2. **不影响质量**——LLM 不会因为上限高就多写，生成内容由提示词和模型行为决定
3. **部分模型有上下文窗口限制**，`max_tokens` 不能超过 `context_window - prompt_tokens`

**建议：** 对 RAG 场景，`1024~2048` 是比较合适的范围。如果 prompt 本身很长（检索了 10 个长片段），需要确保 `max_tokens + prompt_tokens < model_context_limit`。

---

### Q3: 怎么用 `$env:OPENAI_API_KEY` 切回 OpenAI？优先级由哪行决定？

**切回 OpenAI 的方式：** 同时清掉 DeepSeek key，设置 OpenAI key：

```powershell
$env:DEEPSEEK_API_KEY = $null
$env:OPENAI_API_KEY = "sk-your-openai-key"
python main.py ask "你的问题"
```

**优先级由 `generator.py` 第 59-64 行 `_detect_provider` 方法决定：**

```python
def _detect_provider(self) -> dict | None:
    for name in ["deepseek", "openai"]:  # ← 先遍历 deepseek，再遍历 openai
        cfg = PROVIDERS[name]
        if os.environ.get(cfg["env_key"]):
            return cfg   # ← 第一个找到的就返回
    return None
```

循环顺序 `["deepseek", "openai"]` 决定了 DeepSeek 优先。反过来改成 `["openai", "deepseek"]` 就是 OpenAI 优先。

也可以扩展这个字典，比如加入本地 Ollama：

```python
"ollama": {
    "env_key": "OLLAMA_HOST",
    "base_url": "http://localhost:11434/v1",
    "model": "qwen2.5:7b",
    "label": "LLM (Ollama本地)",
},
```
