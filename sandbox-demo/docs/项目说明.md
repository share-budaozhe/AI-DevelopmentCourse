# 安全沙箱应用 Demo —— 项目说明

## 1. 项目背景

在 AI 应用（特别是 AI Agent）中，LLM 生成的代码需要在安全环境中执行。以下是真实的风险场景：

- **AI Agent 调用工具时**：LLM 生成的参数可能包含恶意代码
- **代码解释器场景**：用户或 LLM 生成的 Python 代码需要隔离执行
- **多租户平台**：不同用户的代码不能相互影响
- **安全审计要求**：所有代码执行必须有完整日志

**沙箱（Sandbox）** 技术正是为了应对这些挑战而设计的——让不受信任的代码在受控环境中运行，即使代码是恶意的，也无法对宿主机造成实质性伤害。

## 2. 项目目标

- **展示三种主流沙箱技术**：覆盖进程级、容器级、语言级三个隔离层次
- **提供可复用的沙箱模块**：抽象接口统一，三种实现可插拔替换
- **建立安全策略体系**：AST 静态分析 + 导入控制 + 资源限制 + 审计日志
- **对比适用场景**：帮助开发者根据需求选择最合适的沙箱方案

## 3. 三种沙箱技术

### 3.1 Subprocess 资源限制沙箱

| 维度 | 说明 |
|------|------|
| 隔离级别 | 进程级（Process） |
| 核心技术 | 子进程启动 + OS RLIMIT + SIGALRM 超时 |
| 启动速度 | ~50ms |
| 内存开销 | ~30MB |
| 外部依赖 | 无（纯 Python + subprocess） |
| 跨平台 | ✅ Windows / Linux / macOS |

**工作原理**：
```
主进程                       子进程
  │                            │
  │── subprocess.Popen() ────▶│ 启动
  │                            ├── resource.setrlimit()     设置 CPU/内存/进程限制
  │                            ├── signal.signal(SIGALRM)   设置超时
  │                            ├── signal.signal(SIGXCPU)   设置 CPU 超时
  │                            ├── 重定向 sys.stdout/stderr
  │                            ├── exec(code)               执行代码
  │                            ├── 收集输出
  │                            │
  │── communicate(timeout) ───▶│ 等待完成
  │◀── JSON 结果 ──────────────│ 返回
  │                            │
  │ (超时则 kill)               │
```

**适用场景**：
- LLM Agent 执行生成的代码片段
- 可信用户的批量计算任务
- 需要精细资源控制的场景

### 3.2 Docker 容器沙箱

| 维度 | 说明 |
|------|------|
| 隔离级别 | 容器级（Container） |
| 核心技术 | Docker SDK + Cgroup + Namespace + Capabilities |
| 启动速度 | ~1s |
| 内存开销 | ~50MB |
| 外部依赖 | Docker Engine / Docker Desktop |
| 跨平台 | ⚠️ 需要 Docker 环境 |

**安全加固层**：
```
Layer 1: Linux Namespace (PID/NET/MNT/IPC/UTS/USER)
Layer 2: Cgroup (cpu_quota / mem_limit / pids_limit)
Layer 3: Capabilities (cap_drop=ALL + no-new-privileges)
Layer 4: Filesystem (read_only rootfs + tmpfs /tmp)
Layer 5: Network (network_mode=none)
Layer 6: Ulimit (nproc / nofile / fsize)
Layer 7: Python 级 RLIMIT (容器内第二道防线)
Layer 8: Auto-remove (容器即焚，不留痕迹)
```

**适用场景**：
- 执行不受信任的第三方代码
- 多租户隔离要求严格的平台
- 需要完全隔离的安全敏感场景

### 3.3 RestrictedPython AST 沙箱

| 维度 | 说明 |
|------|------|
| 隔离级别 | 语言级（Language） |
| 核心技术 | AST 变换 + 受限内置函数 + 导入白名单 |
| 启动速度 | < 1ms |
| 内存开销 | ~5MB |
| 外部依赖 | 无（纯 Python） |
| 跨平台 | ✅ 全平台 |

**限制机制**：
```
代码输入
  │
  ▼
AST 解析 ──────── 语法检查
  │
  ▼
AST 变换 ──────── 危险节点替换
  │
  ▼
builtins 替换 ──── eval/exec/open 等被安全版本替换
  │
  ▼
导入检查 ──────── 仅允许白名单模块
  │
  ▼
compile() ─────── 编译为字节码
  │
  ▼
exec() ────────── 在受限 globals 中执行
```

**适用场景**：
- 教学平台的 Python 练习环境
- 用户自定义公式/规则引擎
- 数据转换/ETL 中的自定义逻辑
- 纯计算型代码执行

## 4. 沙箱选择决策树

```
需要执行不受信任的 Python 代码
    │
    ├── 代码来自外部/第三方？
    │   ├── 是 → Docker 容器沙箱（最强隔离）
    │   └── 否 → 继续判断
    │
    ├── 需要访问网络/文件？
    │   ├── 需要 → Docker 容器沙箱
    │   └── 不需要 → 继续判断
    │
    ├── 执行频率高，需要低延迟？
    │   ├── 是 → Subprocess 或 RestrictedPython
    │   └── 否 → Docker 容器沙箱
    │
    ├── 只需要纯数学计算？
    │   ├── 是 → RestrictedPython AST 沙箱（最快）
    │   └── 否 → Subprocess 资源限制沙箱
    │
    └── 需要跨平台且无 Docker？
        ├── 是 → Subprocess 或 RestrictedPython
        └── 否 → Docker 容器沙箱
```

## 5. 项目价值

1. **教育价值**：通过可运行的代码理解三种沙箱原理
2. **参考价值**：提供安全策略设计的最佳实践
3. **复用价值**：抽象接口设计，可直接集成到实际项目中
4. **对比价值**：同一组测试用例在三种沙箱上运行，直观对比
